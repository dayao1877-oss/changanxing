## 1. 命名规范

### 1.1 通用命名规则
- **所有命名必须使用拼音**
- **所有命名必须以sj开头**（司机拼音首字母）
- 采用下划线命名法（snake_case）
- 命名要清晰表达含义，避免缩写

### 1.2 数据库命名

#### 表名
- 格式：`sj_xxx`
- 示例：
  - `sj_siji`（司机表）
  - `sj_chengke`（乘客表）
  - `sj_dingdan`（订单表）

#### 字段名
- 格式：`sj_xxx`
- 示例：
  - `sj_id`（主键ID）
  - `sj_xingming`（姓名）
  - `sj_shouji`（手机）
  - `sj_zhuangtai`（状态）
  - `sj_chuangjian_shijian`（创建时间）
  - `sj_gengxin_shijian`（更新时间）

### 1.3 API接口命名

#### 接口路径
- 格式：`/api/sj/xxx`
- 示例：
  - `/api/sj/denglu`（登录）
  - `/api/sj/dingdan/list`（订单列表）
  - `/api/sj/daka`（打卡）

#### 接口函数名
- 格式：`sjXxx`
- 示例：
  - `sjDenglu()`
  - `sjDingdanList()`
  - `sjDaka()`

### 1.4 函数/方法命名

#### 前端函数（Vue）
- 格式：`sjXxx`（驼峰命名）
- 示例：
  - `sjGetDingdanList()`
  - `sjHandleDaka()`
  - `sjSubmitData()`

#### 后端函数（TypeScript/Nuxt）
- 格式：`sjXxx`（驼峰命名）
- 示例：
  - `sjCheckAuth()`
  - `sjQueryDingdan()`
  - `sjSaveDaka()`

### 1.5 组件命名（Vue）
- 格式：`SjXxx`（大驼峰命名）
- 示例：
  - `SjDingdanCard`
  - `SjDakaButton`
  - `SjMapView`

### 1.6 变量命名
- 格式：`sjXxx`（驼峰命名）
- 示例：
  - `sjDingdanList`
  - `sjCurrentUser`
  - `sjLoading`

## 2. 模块化设计规范

### 2.1 模块独立原则
- 每个模块功能单一，职责明确
- 模块之间通过接口调用，避免直接依赖
- 模块内部高内聚，模块之间低耦合

### 2.2 模块划分

#### 核心模块
- `sj-auth`：认证授权模块
- `sj-user`：用户管理模块（司机、乘客）
- `sj-order`：订单管理模块
- `sj-checkin`：打卡管理模块
- `sj-location`：定位管理模块
- `sj-exception`：异常管理模块
- `sj-sos`：紧急求助模块
- `sj-finance`：财务管理模块
- `sj-system`：系统管理模块

#### 工具模块
- `sj-utils`：通用工具函数
- `sj-storage`：存储工具
- `sj-http`：HTTP请求封装
- `sj-map`：地图工具

### 2.3 模块调用规范
- 模块必须提供清晰的公开API
- 内部实现细节对外不可见
- 通过依赖注入或模块导入方式调用
- 避免循环依赖

## 3. 文档规范

### 3.1 文档要求
- **每个功能模块完成后必须编写对应文档**
- 文档保存在 `docs/` 目录下
- 文档格式统一使用Markdown

### 3.2 文档类型

#### 模块设计文档
- 文件名：`模块名-设计文档.md`
- 内容包括：
  - 模块功能说明
  - 模块架构图
  - API接口列表
  - 数据结构定义
  - 依赖关系说明

#### API接口文档
- 文件名：`模块名-API文档.md`
- 内容包括：
  - 接口列表
  - 每个接口的请求参数
  - 响应格式
  - 错误码说明
  - 使用示例

#### 数据库表文档
- 文件名：`数据库设计文档.md`
- 内容包括：
  - 表结构说明
  - 字段定义
  - 索引说明
  - ER图

#### 测试文档
- 文件名：`模块名-测试文档.md`
- 内容包括：
  - 测试用例
  - 测试步骤
  - 预期结果
  - 测试记录

## 4. 测试规范

### 4.1 测试要求
- **每个功能模块完成后必须立即进行测试**
- 测试前编写测试用例
- 测试通过后才能进入下一阶段

### 4.2 测试类型

#### 单元测试
- 测试每个独立函数/方法
- 覆盖率要求：核心功能≥80%

#### 接口测试
- 测试API接口的正确性
- 测试参数校验
- 测试错误处理

#### 集成测试
- 测试模块之间的协作
- 测试完整业务流程

#### 端到端测试
- 测试完整用户场景

### 4.3 测试流程
1. 编写测试用例
2. 执行测试
3. 记录测试结果
4. 修复Bug
5. 回归测试
6. 测试通过，编写测试文档

## 6. Git提交规范

### 6.1 提交信息格式
```
<type>(<scope>): <subject>

<body>

<footer>
```

### 6.2 Type类型
- `feat`：新功能
- `fix`：修复Bug
- `docs`：文档更新
- `style`：代码格式调整
- `refactor`：重构
- `test`：测试相关
- `chore`：构建/工具相关

### 6.3 示例
```
feat(sj-order): 添加订单创建功能

- 实现订单创建API
- 添加订单创建表单组件
- 编写单元测试

Closes #123
```
